# 1.2.4 线程本地存储（TLS）— glibc 中的实现与用法

本文用「小白能懂」的方式介绍：TLS 是什么、为什么需要、在 glibc 里怎么实现、怎么用。

---

## 一、先打个比方：为什么需要「每线程一份」的变量？

想象一家公司里：

- **全局变量** = 公司只有一块白板，所有人都往上面写。谁改了什么，别人都会看到，容易写乱、写丢。
- **TLS（线程本地存储）** = 公司给每个员工发一个**带锁的柜子**。大家都有「自己的柜子」，名字都叫「我的柜子」，但里面放的东西各管各的，互不干扰。

在多线程程序里：

- 所有线程**共享同一份内存**（同一块地址空间）。
- 若是**普通全局变量**，所有线程读写的都是**同一块内存**，一个线程改了，别的线程立刻看到，这就是「共享」—— 想要正确，就得加锁、很麻烦，也容易出错。
- **TLS** 的意思是：虽然变量名字在代码里只有一个（比如 `errno`），但**每个线程在运行时都有一份自己的拷贝**。线程 A 改自己的 `errno`，不会影响线程 B 的 `errno`。

所以可以一句话记：**TLS = 名字是全局的，存的数据是「每个线程各有一份」的。**

---

## 二、TLS 是什么？干什么用？

**TLS**（Thread Local Storage，线程本地存储）是一种机制，让程序里声明的变量**按线程分开存**：

- 在**代码里**你只写一个变量名（比如 `__thread int my_id;`）。
- 在**运行时**，每个线程访问这个变量时，拿到的都是**自己那一份**，不会和别的线程混在一起。

**主要作用可以概括为：**

1. **线程私有数据**：每个线程需要自己的「副本」，而不是共享一份。例如：当前线程的 errno、当前线程的 locale、当前线程的随机数状态。
2. **避免锁**：数据本来就是每线程一份，不需要互斥锁，性能好、代码简单。
3. **隐式传参**：有些「上下文」不用在每个函数参数里传来传去，而是放在 TLS 里，当前线程随时取用（例如「当前请求 ID」「当前用户」）。

glibc 里很多地方都依赖 TLS，比如：

- **errno**：每个线程有自己的 errno，一个线程的系统调用错误不会覆盖另一个线程的。
- **locale**：当前线程的 locale 设置存在 TLS 里。
- 内部实现里也会用 TLS 做**每线程缓存**（例如和 malloc 的 tcache 思路类似：每线程一块小缓存，减少竞争）。

## 三、为什么引入 TLS？

在没有 TLS 的年代，要「每线程一份数据」只能：

- **方案 A**：用**全局数组**，下标是线程 ID。  
  问题：要事先知道最大线程数、要管理生命周期、访问时总要带一个「线程 ID」，麻烦且易错。
- **方案 B**：用 **pthread_key**（pthread_getspecific / pthread_setspecific）。  
  可以按线程存一块「自定义数据」，但每次访问都要**查表**，接口是「键 + 取/设」，不如直接写变量名自然；而且要自己管理 key 和内存。

引入 TLS 的目的就是：

- **语言/编译器层面**支持「每线程一个变量」：写起来像普通全局变量，用起来每个线程自动拿到自己的那份。
- **不需要**在代码里写「根据线程 ID 去数组里取」或「pthread_getspecific(key)」，编译器与运行时（glibc + 动态链接器）帮你把「这个名字」映射到「当前线程的那块内存」。
- **性能更好**：最终变成「从当前线程的 TLS 块里按固定偏移取」，比查表快，也方便 CPU 和操作系统用专门机制（如 x86-64 的 `%fs`）优化。

所以：**引入 TLS = 让「每线程一份变量」变成语言级、编译器级支持，既好写又高效。**

## 四、在 C 里怎么用？（小白用法）

### 1. 用 `__thread`（编译器直接支持，推荐）

在变量前加 `__thread`，这个变量就变成「每个线程各有一份」：

```c
#include <stdio.h>
#include <pthread.h>

__thread int my_thread_id;   // 每个线程都有自己的 my_thread_id

void* thread_func(void* arg) {
    my_thread_id = (int)(long)arg;  // 只改自己的那份
    printf("thread %d: my id = %d\n", (int)(long)arg, my_thread_id);
    return NULL;
}

int main() {
    pthread_t t1, t2;
    pthread_create(&t1, NULL, thread_func, (void*)1);
    pthread_create(&t2, NULL, thread_func, (void*)2);
    pthread_join(t1, NULL);
    pthread_join(t2, NULL);
    return 0;
}
```

- 两个线程都会打印出**自己**赋的 id，不会串。
- 你**不用**写「根据 pthread_self() 去某个数组里取」—— 编译器与 glibc 会生成「从当前线程的 TLS 里读」的代码。

**注意**：`__thread` 是 GCC/Clang 的扩展，C11 标准里等价写法是 `_Thread_local`（要 `#include <threads.h>` 或至少声明）。glibc 在 Linux 上配合 GCC 主要用 `__thread`。

### 2. 用 pthread_key（不扩展、可动态创建「每线程一块数据」）

若不想用编译器扩展，或需要「运行时再决定存什么」，可以用 POSIX 的 **pthread_key**：

```c
pthread_key_t key;

void init_key() {
    pthread_key_create(&key, NULL);  // 创建一个「每线程槽位」
}

// 在任意线程里：
pthread_setspecific(key, 某块内存);   // 给当前线程存一份
void* p = pthread_getspecific(key);   // 取出当前线程存的那份
```

- 每个线程通过同一个 `key` 存/取自己的那一份指针（或小数据）。
- 和 `__thread` 比：更灵活（可以存任意指针），但每次访问是「查表」，不如 `__thread` 快；而且要自己管理 key 和内存生命周期。

glibc 内部有些历史代码或需要动态注册「每线程数据」的场景，会用到类似 **TSD**（Thread-Specific Data）的接口；而新代码和 C 层推荐用 `__thread`。

## 五、glibc 里 TLS 是怎么实现的？（简化版，小白能懂）

下面用「人话」说个大概，不抠细节。

### 1. 每个线程有一块「自己的 TLS 区域」

- 操作系统/glibc 在**创建线程**时，会为这个线程分配一块**只有它自己能用的内存**，专门放「线程本地变量」。
- 这块内存里会按**固定布局**放：
  - 一些**内部用的控制信息**（线程描述、动态链接用的小表等）；
  - 你声明的各个 `__thread` 变量，每个占一个**固定偏移**。

可以理解为：每个线程有一个「小柜子」，柜子里第几格放哪个变量，在链接/加载时就定好了。

### 2. 当前线程怎么找到自己的「柜子」？—— 线程指针（TP）

- CPU/ABI 约定：**当前线程**有一个「线程指针」（Thread Pointer），总是指向**自己**那块 TLS 区域的某个位置。
- 在 **x86-64 Linux** 上，这个指针放在 **段寄存器 `%fs` 的基址**里（`fs.base`）。  
  所以：**读「当前线程的 TLS 变量」= 用 `%fs` 做基址 + 某个偏移**。
- 在 **AArch64** 上，用的是 **TPIDR_EL0** 寄存器。

这样，访问 TLS 不需要「先查我是哪个线程再查表」—— 一条指令就能从 `%fs:偏移` 取到当前线程的那份数据，所以快。

### 3. 程序/库里有几个 `__thread` 变量，怎么知道偏移？

- 编译时：编译器把 `__thread` 变量放进 **ELF 的 TLS 段**（如 `.tdata`、`.tbss`），并生成「用某个偏移访问」的指令。
- 链接时：链接器会算好**每个模块（可执行文件、每个 so）**里 TLS 变量合在一起占多大、每个变量在「TLS 块」里的偏移是多少。
- ELF 里有一种 **PT_TLS** 的程序头，描述「TLS 镜像」的大小、对齐等。  
  动态链接器（ld.so）和 glibc 在**创建线程**时，会按 PT_TLS 的说明为这个线程分配一块内存，把 TLS 的初始值拷进去，并把这块内存的基址和线程指针（如 `%fs`）关联起来。

所以：**你写 `__thread int x;`，编译器与链接器负责算出「x 在 TLS 块里的偏移」，运行时每个线程的 TLS 块里都在这个偏移处放一个 `int`。**

### 4. 动态加载的 .so 也有 `__thread` 怎么办？—— DTV

- 主程序和已经加载的 .so 的 TLS 合在一起，在**线程创建时**就分配好，叫 **Static TLS**。
- 若后来用 `dlopen` 打开了带 `__thread` 的 .so，就需要给**已经存在的线程**补一块 TLS（**Dynamic TLS**）。
- glibc 用 **DTV**（Dynamic Thread Vector）来记录「每个已加载模块的 TLS 块」在当前线程里的位置。  
  访问动态加载模块里的 TLS 时，先通过 DTV 找到该模块的 TLS 块，再按偏移取——比 Static TLS 多一次间接寻址，但语义一致。

小白只需记住：**静态链接/主程序里的 `__thread` 最快（直接 %fs + 偏移）；动态加载的 .so 里的 `__thread` 稍慢一点，但一样是「每线程一份」。**

## 六、和 glibc 其他部分的关系（建立整体图景）

- **NPTL（线程库）**：线程创建、销毁时，glibc 会分配/释放该线程的 TLS 区域，并设置好线程指针（如 `%fs`）。所以 TLS 是「每线程」的，和 NPTL 紧密相关。
- **动态链接器（ld.so）**：负责解析 PT_TLS、在进程/线程初始化时布置 TLS 布局，并配合 glibc 的 NPTL 设置线程指针。
- **errno、locale 等**：glibc 里很多「每线程一个」的全局状态，就是用 TLS 或类似机制实现的，这样多线程下不会串。

---

## 七、常见使用场景（帮你对上号）

| 场景 | 说明 |
|------|------|
| **errno** | 每个线程自己的错误码，glibc 用 TLS 存。 |
| **locale** | 当前线程的 locale，也是 TLS。 |
| **「当前请求/用户」** | 在 Web 服务里，把 request_id、user 等放进 TLS，处理函数不用层层传参。 |
| **非线程安全的单例** | 例如每个线程要一个「自己的」随机数生成器、日期格式化器，用 `__thread` 包一层即可。 |
| **性能敏感的每线程缓存** | 类似 malloc 的 tcache：每线程一小块缓存，减少锁竞争。 |

## 八、小结（背这几句就够）

- **TLS** = 线程本地存储，让一个「全局名字」的变量，在运行时**每个线程各有一份**，互不干扰。
- **为什么引入**：多线程下要「每线程独立数据」时，不用全局数组 + 线程 ID，也不用每次查 pthread_getspecific；编译器与 glibc 直接支持，又好写又快。
- **在 C 里怎么用**：加 `__thread`（或 C11 的 `_Thread_local`）即可；需要动态「每线程一块」时用 pthread_key。
- **glibc 里怎么实现**：每个线程一块 TLS 区域，用线程指针（x86-64 上是 `%fs`）加固定偏移访问；ELF 有 PT_TLS 描述布局；动态加载的 .so 用 DTV 扩展。
- **典型用途**：errno、locale、每线程上下文、每线程缓存/对象，避免锁、简化接口。

如果你刚学多线程，只要记住：**想要「每个线程一份、互不干扰」的变量，用 `__thread` 就对了；glibc 和系统在背后帮你把「这个名字」映射到「当前线程的那块内存」。**
