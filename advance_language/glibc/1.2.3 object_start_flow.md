# 1.2.3 程序启动与「对象」初始化流程 — glibc 里发生了什么

本文用「小白能懂」的方式讲清楚：**你运行一个 C/C++ 程序时，在 `main()` 被调用之前，系统到底做了哪些事？** 这些事在 glibc 里就是「程序与库的初始化流程」，也可以理解为**各种「对象」和状态在进入 main 前的准备过程**。

---

## 一、一个最核心的问题：谁在调用 main？

你可能以为：**我执行 `./a.out`，操作系统就直接跑我写的 `main()`。**

其实不是。**操作系统只会跳到程序里约定好的「入口」**，这个入口一般**不是** `main`，而是一个叫 **_start** 的小函数。  
真正调用 `main()` 的是 **glibc** 提供的启动代码：它先做完一堆「准备工作」，再调用你的 `main`。

所以可以记一句：**main 不是程序的真正入口，_start 才是；glibc 在 _start 和 main 之间插了一整套「初始化流程」。**

---

## 二、为什么要做「初始化」？—— 这些「对象」从哪来？

程序里不只有 `main` 里的代码，还有：

- **全局变量、静态变量**：C 里可能带初始值（包括用函数赋的初值）；C++ 里还有**全局对象**，它们的**构造函数**必须在用之前执行。
- **库的准备工作**：例如 C 标准库要准备好 **stdin / stdout / stderr**，线程库要初始化一次，等等。
- **你写的「在 main 之前跑」的代码**：比如用 `__attribute__((constructor))` 声明的函数，或 C++ 里全局对象的构造函数。

这些都可以笼统地看成「**程序与库在进入 main 之前必须准备好的对象与状态**」。  
**对象初始化流程** = 在调用 `main()` 之前，按约定顺序把这些东西都准备好。

---

## 三、整体流程：从「入口」到 main，再到退出

可以简化为一条线（先不用管细节）：

```
内核加载程序
    ↓
_start（真正的入口）
    ↓
__libc_start_main（glibc 的「总控」）
    ↓
  ├─ 做各种一次性初始化
  ├─ 执行 .preinit_array（若存在）
  ├─ 执行 _init（里面可能再调 .ctors / 旧式构造函数）
  ├─ 执行 .init_array（新式构造函数、__attribute__((constructor))、C++ 全局构造）
  ↓
main(argc, argv)  ← 你的代码终于在这里跑
    ↓
main 返回 或 调用 exit()
    ↓
__libc_start_main 的「收尾」：
  ├─ 执行 atexit 注册的函数
  ├─ 执行 .fini_array（析构、收尾）
  ├─ 执行 _fini
  ├─ rtld_fini（动态链接器收尾）
  ↓
_exit() 系统调用，进程结束
```

下面分块说每个环节在「干什么」、有什么「作用」。

---

## 四、各环节是干什么的？（小白版）

### 1. _start：程序真正的入口

- **谁提供的？** 一般是**编译器/链接器**带来的一个小目标文件（如 **crt1.o**）里的代码，不是你自己写的。
- **干什么？** 做**最少必要**的事：准备好栈、**argc / argv / 环境变量**等，然后去调 **__libc_start_main**，并把「main 的地址」「init」「fini」等传进去。
- **为什么不能直接调 main？** 因为在那之前，C 运行环境还没准备好（例如全局变量没初始化、stdio 没准备好），直接进 main 会出错。

可以理解为：**_start 只负责「把摊子摆好，然后交给 glibc」。**

### 2. __libc_start_main：glibc 的「总控」

这是 **glibc** 里的函数，负责「从 _start 到 main，再从 main 到进程退出」的整条链路。

**在调用 main 之前，它大致会：**

1. **栈、安全相关**：栈保护、安全校验等（视实现而定）。
2. **注册退出时要做的事**：把 **atexit**、**fini**、**rtld_fini** 等登记好，这样 main 返回或你调 exit() 时，会按顺序做收尾。
3. **跑「构造函数」链**：也就是我们说的**对象初始化流程**，顺序大致是：
   - **.preinit_array**（只有可执行文件本身有，且一般在「依赖的 .so 的 init 之前」跑，用于最早的一批初始化）；
   - **_init**：老式约定，一个「初始化函数」；里面可能再调 **.ctors** 或由 crt 安排的逻辑（例如性能分析用的 __gmon_start__）；
   - **.init_array**：新式约定，一个「函数指针数组」，里面是各个构造函数（包括 `__attribute__((constructor))`、C++ 全局对象的构造函数等），**按数组顺序依次调用**。
4. **调用 main(argc, argv)**。

**main 返回或你调用 exit() 之后：**

- 按登记顺序执行 **atexit** 的回调；
- 执行 **.fini_array**（顺序一般与 .init_array **相反**，相当于「析构」）；
- 执行 **_fini**；
- 执行 **rtld_fini**（动态链接器收尾）；
- 最后 ** _exit()** 把进程结束掉。

所以：**「对象初始化」= __libc_start_main 里在 main 之前跑的那一串（preinit_array → _init → init_array）；「对象收尾」= main 之后跑的那一串（atexit → fini_array → _fini → rtld_fini）。**

### 3. .preinit_array：最早的一批初始化

- **是什么？** ELF 里的一段「函数指针数组」，**只在可执行文件里有效**，动态库里的会被忽略。
- **作用？** 在**依赖的 .so 的初始化之前**先跑，适合「主程序必须最早做的准备」。
- **谁往里面放？** 一般要手写或通过链接脚本；普通 C/C++ 代码更常用的是下面的 .init_array 和 constructor。

### 4. _init 与 .ctors（老式机制）

- **_init**：一个单独的「初始化函数」的入口，地址写在 ELF 的 **DT_INIT** 里。通常由 **crti.o + crtbegin.o + crtn.o** 拼成：里面会调 **__gmon_start__**（做性能分析用，如 gcc -pg），以及老式的 **.ctors** 里的构造函数。
- **.ctors**：老式的「构造函数列表」；执行顺序和 .init_array 略有不同（.ctors 是「逆序」跑），现在新代码多用 .init_array。

小白只需知道：**老程序/老库可能还用 _init/.ctors；新的一般用 .init_array。** glibc 两套都支持。

### 5. .init_array：新式「构造函数」列表（最常用）

- **是什么？** ELF 里的 **DT_INIT_ARRAY**，一个「函数指针数组」。
- **里面有什么？**
  - 用 **`__attribute__((constructor))`** 声明的函数；
  - C++ 的**全局对象**的构造函数（编译器会往这里塞）；
  - 带优先级的还有 **.init_array.101**、**.init_array.102** 等（数字越小越先执行；0–100 一般保留给实现，应用用 101–65535）。
- **作用？** 在 main 之前**按顺序**把这些函数都跑一遍，相当于「所有要在 main 前完成的初始化」都集中在这里。
- **谁在跑？** 动态链接器（ld.so）加载完可执行文件和 .so 之后，会按依赖顺序对每个模块执行它的 DT_INIT_ARRAY；静态链接时则由 __libc_start_main 调用的 init 链里执行。

一句话：**你想「在 main 之前自动跑点什么」，用 `__attribute__((constructor))` 就行，编译器会把它放进 .init_array。**

### 6. .fini_array 与 _fini：程序退出时的收尾

- **.fini_array**：和 .init_array 对应，是「析构/收尾」函数指针数组；**执行顺序一般和 .init_array 相反**（后初始的先收尾）。
- **_fini**：老式的「终止函数」，对应 DT_FINI。
- **何时跑？** main 返回或 exit() 被调用后，在 atexit 之后、进程真正 _exit 之前，由 glibc/ld.so 按约定顺序执行。

---

## 五、和「对象」的关系（C 与 C++）

- **C**：全局/静态变量若带初始值（包括用函数赋初值），编译器可能生成在 .init_array 里调用的初始化函数，在 main 前执行。
- **C++**：全局对象的**构造函数**必须在 main 之前执行，所以编译器会把它们放进 .init_array（或老式的 .ctors）；**析构函数**则在 .fini_array（或 .dtors）里，在 main 之后、进程退出前执行。

所以「对象初始化流程」里的「对象」不单指 C++ 对象，而是**所有「在 main 前必须准备好的程序/库状态」**；C++ 只是其中最典型的一类。

---

## 六、谁提供了 _start 和 .init/.fini 的「壳」？

这些通常来自** C 运行时（CRT）** 的一小组目标文件，由**编译器/工具链**提供，链接时自动加进去：

| 文件（常见名） | 作用（小白版） |
|----------------|----------------|
| **crt1.o**（或类似） | 提供 **_start**，程序入口，准备好参数后调 __libc_start_main。 |
| **crti.o** | 提供 **_init / _fini** 的「开头」一段（例如压栈、调 __gmon_start__）。 |
| **crtn.o** | 提供 _init / _fini 的「结尾」一段（弹栈、返回）。 |
| **crtbegin.o / crtend.o**（GCC） | C++ 用，参与 .ctors/.dtors 或 .init_array/.fini_array 的布局，以及全局构造/析构的调用。 |

你一般**不用**自己管这些文件，只要知道：**入口是 _start，真正张罗初始化和调用 main 的是 glibc 的 __libc_start_main。**

---

## 七、动态链接时：可执行文件和 .so 的初始化顺序

程序若依赖 **b.so** 和 **c.so**，动态链接器（ld.so）会：

1. 先加载这些 .so；
2. 对**每个**模块（可执行文件、每个 .so）按**依赖顺序**执行它的 **DT_INIT_ARRAY** 和 **DT_INIT**（_init）；
3. 可执行文件自己的 **.preinit_array** 会在「依赖的 .so 的 init 之前」执行（具体顺序见 ABI 和 ld.so 实现）。

这样，**库的「对象」和状态会在 main 之前、按依赖关系准备好**；主程序的 .preinit_array 可以抢在最前面。

---

## 八、小结（背这几句就够）

- **main 不是入口**：真正入口是 **_start**，它只做最少准备，然后调 **__libc_start_main**。
- **对象初始化流程** = 在 **main 之前**按顺序执行：**.preinit_array**（若有）→ **_init**（内部可能跑 .ctors）→ **.init_array**（构造函数、`__attribute__((constructor))`、C++ 全局构造）。这些由 glibc 和 ld.so 按 ELF 约定完成。
- **作用**：让全局/静态变量、C++ 全局对象、库的一次性设置，都在 main 之前就绪；退出时再按 **.fini_array → _fini** 等做收尾。
- **为什么引入**：C/C++ 语义要求「在 main 之前完成全局初始化」；ELF 和 glibc 用 .preinit_array / _init / .init_array 来满足这个要求，并统一库和主程序的启动顺序。
- **小白用法**：想「在 main 之前跑一段代码」就写 **`__attribute__((constructor))`**，想「进程退出时跑」就写 **`__attribute__((destructor))`** 或 **atexit**。

一句话：**glibc 的「对象初始化流程」就是把「程序与库在 main 前该准备好的东西」按约定顺序跑一遍，保证你进 main 时，全局对象和库状态都已经就绪。**
