# 1.2.1 malloc 机制

glibc 默认 malloc 实现是 **ptmalloc2**（pthreads malloc），源自 **dlmalloc**（Doug Lea malloc），经多线程 arena 扩展而成，属于「堆式」分配器：在更大的 heap 区域内管理不同大小的 chunk，支持多 heap、多 arena。

主要术语（[MallocInternals](https://sourceware.org/glibc/wiki/MallocInternals)）：

- **Arena**：被一个或多个线程共享的结构，包含若干 heap 的引用及各类 free chunk 链表；线程从所属 arena 的空闲链表分配。
- **Heap**：一块连续内存区，被划分成 chunk；每个 heap 只属于一个 arena。
- **Chunk**：可分配（归应用）、可释放（归 glibc）、可与相邻 chunk 合并的小块；每个 chunk 位于一个 heap、属于一个 arena。

## 整体架构

```
用户态 malloc → __libc_malloc → arena_get() → tcache(thread local) → fastbin → smallbin → largebin → unsorted bin → top chunk → sysmalloc() → brk/mmap → 内核
```

## 核心概念

### Chunk

malloc 以 **chunk** 为单位管理内存：heap 被切成若干 chunk，每个 chunk 带元数据（本块大小、前一块大小等）。**用户拿到的指针是 payload 起始地址**；内部用 `mchunkptr` 指向「本 chunk 的起始」——在实现上，**mchunkptr 指向的是「前一个 chunk 的最后一个字」**，即当前 chunk 的第一个字段 `prev_size` 只有在「前一块空闲」时才有意义。

**已用 chunk** 只保留 size 信息；**空闲 chunk** 会复用 payload 区存链表指针等，且**块尾会存一份 size 副本**（低 3 位清零，用于边界与合并）。

因 chunk 大小按 8 字节对齐，**size 的低 3 位用作标志位**：

| 标志 | 值 | 含义 |
|------|---|------|
| **P (Previous in use)** | 0x01 | 前一个 chunk 仍在使用 → `prev_size` 无效；也用于避免与 fastbin 等中的块合并。 |
| **M (MMap'd)** | 0x02 | 本块由单次 `mmap` 分配，不属于任何 heap。 |
| **A (Non-main arena)** | 0x04 | 非主 arena（主 arena 用进程初始 heap）；为 1 表示来自 mmap 的 heap，便于由 chunk 地址推算 heap_info。 |

最小 chunk 大小为 `4*sizeof(void*)`（或 ABI 要求的更大对齐），以保证 payload 能放下后续管理所需开销。

### Arena

一个 **arena** 包含：

- 一个或多个 **heap** 的引用；
- 该 arena 上的 **free chunk 链表**（fastbins、unsorted bin、smallbins、largebins）；
- **top chunk**（通常为当前最大可用块，多与最近扩展的 heap 相邻）；
- **arena mutex**，除 fastbin 等可用原子操作外，其余操作需加锁。

**主 arena**：对应进程**初始 heap**（紧接 .bss 等），有静态变量指向；**非主 arena** 的 heap 通过 **mmap** 分配。arena 数量上限默认为 **8 × CPU 核数**（可由 [mallopt(3)](https://man7.org/linux/man-pages/man3/mallopt.3.html) 等调节），以在竞争与碎片之间折中。

### Heap

**Heap** 是一块连续、可扩展的内存区：

- **主 arena** 的 heap 由 **brk/sbrk** 扩展；
- **非主 arena** 的 heap 通过 **mmap** 新增，用尽可再 mmap 新 heap。

heap 按 2 的幂对齐，因此对「来自已分配 heap」的 chunk（A 位可区分），可从 chunk 地址推算出其所属的 `heap_info`。

### Bins（空闲链表）

空闲 chunk 按大小与用途放入不同 **bin**：

| 类型 | 说明 |
|------|------|
| **fastbins** | 小尺寸、按尺寸分档的单向链表；**不合并**相邻块，逻辑简单、访问快；需要时可被移入其他 bin。 |
| **unsorted bin** | free 时先丢进**唯一**的 unsorted bin，在 **malloc 时再被分类**到 small/large，并有机会被直接复用，排序逻辑集中在一处。 |
| **smallbins** | 每档尺寸固定，双链表；放入前会与相邻空闲块 **coalesce**，故不会与同 bin 内块相邻。 |
| **largebins** | 每档对应一个尺寸**范围**，需在 bin 内找「足够大」的块，可能 **split**；除 fd/bk 外还有 **fd_nextsize / bk_nextsize** 按尺寸排序的链，便于快速找「第一个足够大」的块（通常选同尺寸中的第二块，以少动 nextsize 链）。 |

注：bin 本身不是 chunk，实现上用与 chunk 重叠的 fwd/bck 布局，使 `mchunkptr` 的 fd、bk 能访问对应 bin 的链表。

### tcache（Thread Local Cache）

glibc 2.26+ 引入的 **每线程缓存**：

- 每线程一个 tcache，**无需锁**即可分配/释放小对象；
- 按 size 分档的**单向链表数组**，链的是 **payload 指针**（与 fastbin 不同）；
- 每档有容量上限（`tcache_count`）；某 size 的 tcache 为空时**不会用更大 size 的块**，而是回退到带锁的 arena 流程，避免内部碎片。

### 小结：Arena 里有什么

- fastbins / unsorted bin / smallbins / largebins  
- top chunk  
- arena mutex、所属线程信息等  
- 一个或多个 heap（主 arena 用 brk heap，非主用 mmap heap）

## malloc 分配算法

（与 [MallocInternals](https://sourceware.org/glibc/wiki/MallocInternals) 一致，顺序如下：）

1. **tcache**：若存在**尺寸完全匹配**的 tcache 块，直接返回，不尝试用更大 bin。
2. **大块 mmap**：请求足够大时直接用 **mmap** 向系统要一块（阈值可调，见下平台常量）；free 时对应 **munmap**。
3. **fastbin**：对应 size 的 fastbin 有块则用，并可能从该 fastbin **回填 tcache**。
4. **smallbin**：对应 smallbin 有块则用，也可回填 tcache。
5. **大请求时**：先把 fastbins 中块合并进 unsorted bin。
6. **unsorted bin**：逐个取出，能合并的合并，再按尺寸归入 small/large bin；若遇到**合适大小**的块则直接使用。
7. **large bin**：若请求属「大」尺寸，在对应及更大的 large bin 中找**第一个足够大**的块，必要时 split。
8. 若仍有 fastbin 未合并（小请求可能未做步骤 5），则合并后重复 6、7。
9. **top chunk**：从 top 切一块，不足则 **sysmalloc**（扩展 heap 或 mmap）再切。

**对齐分配**（valloc / pvalloc / memalign）：先按上述流程拿一块「足够大」的块，再按对齐要求切分，多余部分放回 unsorted bin。

## free 算法

1. **tcache**：若该 size 的 tcache 未满，放入 tcache 并返回。
2. **fastbin**：小块放入对应 **fastbin**（不合并）。
3. **mmap 块**：若为单次 mmap 分配的块，直接 **munmap**。
4. **合并**：若与相邻空闲块相邻，则 **coalesce** 成一块。
5. **放入 unsorted bin**：合并后的块放入 unsorted bin（若成为新的 **top chunk** 则不再入 bin）。
6. **大块释放时**：可能触发 fastbins 合并，并检查 **top chunk** 是否足够大以便 **unmap 部分还给系统**（该步骤也可能延迟到后续 malloc 等调用中执行）。

注意：一般情况下 **free 不会把内存还给操作系统**，只是标记为可复用；仅当 top chunk 因释放变得很大时，才可能 unmap 一部分还回内核。

## realloc 算法

- **realloc(NULL, size)** / **realloc(ptr, 0)** 按标准单独处理。
- **mmap 分配的块**：尽量用 **mremap**（若系统支持），否则 malloc-copy-free。
- **arena 堆内块**：
  - **缩小**：若缩小幅度「值得」则 split，前半保留原址返回，后半 free 回 arena；小幅缩小可视为不变。
  - **扩大**：若**下一块**是空闲或 top，且合并后足够大，则合并（必要时再 split），**原指针不变**；否则 **malloc-copy-free**。

## Arena 切换

线程默认绑定一个 arena，但在**分配失败**时（已尝试合并、搜索 free list、处理 unsorted、扩展 heap 等仍无法满足）会尝试 **arena_get_retry**：

- 若当前为非主 arena（mmap heap），可切换到**主 arena**（sbrk heap）；
- 若当前为主 arena，可切换到**非主 arena**（从空闲链表或新建）。

单线程进程可能不做切换直接返回 ENOMEM。低内存时线程可能在高负载下**频繁切换 arena**，以在不同 heap 间寻找空间。

## 堆损坏检测

malloc 子系统中通过 **malloc_printerr** 等做多种检查（如指针对齐、前后向链接等），发现常见错误会终止进程（`__libc_message` → abort）。多数检查是启发式的，伪造得「看起来合法」的 chunk 可能逃过检测；构建时若开启 **NDEBUG**，部分 assert 会被关闭。一旦检测到堆损坏，当前 glibc 不再支持「继续运行」，只能中止。

## ptmalloc 存在的问题

- arena 数量不可控 → 内存膨胀
- 锁竞争仍然存在
- 碎片率较高
- NUMA 友好性差

## 与其他 malloc 实现对比

- **glibc ptmalloc** —“历史演进型，dlmalloc + arena + tcache，兼容与可调性优先”
- **jemalloc** —“碎片与可预测性优先，多级 size class + slab/extent，适合长期运行服务”
- **TCMalloc** — “吞吐与多线程优先，thread cache + central free list + page heap，适合高并发”
- **mimalloc** 等 — 现代通用分配器，代码精简、有界最坏耗时、可选安全特性，可作为 LD_PRELOAD 替代。

| 特性 | glibc (ptmalloc) | jemalloc | TCMalloc |
| --- | --- | --- | --- |
| 多线程 | arena + mutex + tcache | arena + 更细粒度分片 | thread cache + central list |
| 小对象 | fastbin/smallbin 链表 | slab + size class | per-thread free list |
| 大对象 | largebin / mmap | extent tree | page heap (buddy-like) |
| 碎片 | 中等 | 低 | 中等（视负载而定） |
| 兼容性 | 标准 API，默认 | 标准 API | 标准 API |
| 复杂度 | 高 | 高 | 中 |

### jemalloc 要点

- **Size class + slab**：小对象按固定 size class 用 slab 管理，减少碎片与元数据。
- **Extent**：大块用 extent 管理，便于合并与回收。
- **Arena**：多 arena 降低锁竞争，适合多核长运行。

### TCMalloc 要点

- **Thread cache**：小对象优先从线程本地 list 分配/回收，无锁。
- **Central free list**：线程 cache 缺块时从 central 批量取；释放时也可回填。
- **Page heap**：大块按 page 管理，类似 buddy，便于与系统交互。

## 现代 allocator 设计核心主题

### 内存分配算法基础

- First fit / Best fit / Next fit
- Segregated free lists
- Buddy system
- Bitmap / slab / slab+bitmap

| 算法 | 时间复杂度 | 内碎片 | 外碎片 | 并发友好 | 应用 |
| --- | --- | --- | --- | --- | --- |
| First Fit | O(n) | 低 | 高 | 差 | 早期 malloc |
| Best Fit | O(n) | 低 | 高 | 差 | 理论 |
| Next Fit | O(n) | 低 | 高 | 差 | 过渡 |
| Segregated | O(1) | 中 | 中 | 好 | glibc |
| Buddy | O(log n) | 高 | 低 | 好 | OS |
| Bitmap | O(n) | 低 | 低 | 中 | 嵌入式 |
| Slab | O(1) | 低 | 极低 | 极好 | jemalloc |

#### First fit / Best fit / Next fit

在全局单链表中查找：**First fit** 取第一个足够大的块；**Best fit** 取最小足够块；**Next fit** 从上一次查找位置继续。实现简单但易产生碎片、不利于并发。

#### Segregated free lists

按尺寸分档（size class），每档独立链表；分配时先找对应档再取块，常为 O(1)。glibc 的 fastbin/smallbin 即此类。

#### Buddy system

按 2 的幂分块，合并与分裂成对进行；外碎片低，内碎片可能较高，常用于内核页级分配。

#### Bitmap / slab / slab+bitmap

**Slab** 面向对象缓存，同类型对象集中管理，减少碎片与元数据；**Bitmap** 用位图标记块使用情况。jemalloc 等将 slab 与 size class 结合。

**现代 allocator 如何组合这些算法？** 通常按「小 / 中 / 大」对象分层：

- **jemalloc**：小对象 → slab + size class；中对象 → segregated；大对象 → extent tree。
- **TCMalloc**：小对象 → thread free list；中对象 → central free list；大对象 → page heap (buddy-like)。
- **glibc**：小 → fastbin（segregated）+ smallbin；大 → largebin（按尺寸排序）；超大 → mmap。
