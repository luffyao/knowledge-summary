# Itanium C++ ABI 深度学习报告

> 基于 [Itanium C++ ABI](https://itanium-cxx-abi.github.io/cxx-abi/abi.html) 官方文档的详细学习总结。

---

## 一、概述

### 1.1 什么是 C++ ABI

**应用二进制接口（Application Binary Interface, ABI）** 规定了 C++ 程序片段之间、以及程序与实现提供的运行时/库之间的**对象代码接口**。具体包括：

- **数据布局**：C++ 数据对象的内存布局（内置类型、用户定义类型、编译器生成对象如虚表）
- **函数调用约定**：参数传递、返回值、`this`、VTT 等
- **异常处理接口**
- **全局命名（Name Mangling）**
- **各类对象代码约定**

文档最初针对 **Itanium 架构**编写，部分内容仍带有 64 位/Itanium 假设，但设计成可在多种平台上作为通用 C++ ABI 规范使用，通常建立在各平台的 **C ABI** 之上。

### 1.2 文档范围与基础

- **权威性**：本文档不是任何具体平台的“官方”C++ ABI 定义，平台厂商保留最终定义权。
- **基础文档**：依赖 [gABI]（通用 ABI）、[psABI]（Itanium 处理器补充）、[SWCONV]（软件约定与运行时）、[ABI-EH]（异常处理）、C++ 标准等。
- **命名空间与头文件**：扩展 API 声明于 `cxxabi.h`，置于命名空间 `__cxxabiv1`，常用别名 `abi`。

### 1.3 关键术语定义（节选）

| 术语 | 含义 |
|------|------|
| **POD (for layout)** | 用于布局的 POD：一般同 ISO C++ 的 POD，但有例外（如含重叠成员、过宽位域等的结构体不算） |
| **dynamic class** | 需要虚表指针的类（有虚函数或虚基类） |
| **empty class** | 无非静态数据成员（除空类型成员）、无虚函数、无虚基类、无非空非虚直接基类 |
| **nearly empty class** | 仅有虚指针、无其它数据（可无或仅有一个近乎空的非虚直接基类等），可与派生类共享虚指针 |
| **primary base class** | 动态类的、在偏移 0 处与其共享虚指针的唯一直接基类（若存在） |
| **secondary virtual table** | 作为派生类虚表一部分的基类虚表副本 |
| **thunk** | 与目标函数关联的一小段代码，用于在调用前/后调整参数（如 `this`）或环境 |
| **vague linkage** | 具有外部链接但可在多个翻译单元中定义的实体（如 inline、模板）的“单一定义”处理方式 |

### 1.4 限制（64 位 Itanium ABI）

- 非虚基类子对象在完整对象中的**偏移**必须能用 **56 位有符号整数**表示（受 RTTI 实现影响），即类大小有约 2^55 字节的实际上限。

## 二、数据布局（Data Layout）

### 2.1 通用概念

对每个类型，ABI 规定：

- **sizeof(O)**：对象大小  
- **align(O)**：对齐  
- **offset(C)**：各数据成分（基类或成员）在对象 O 内的偏移  

内部还使用：

- **dsize(O)**：数据大小（不含尾部 padding）  
- **nvsize(O)**：非虚大小（不含虚基类）  
- **nvalign(O)**：非虚对齐  

### 2.2 POD 类型

- 布局意义上的 POD 的**大小和对齐**由**基础 C ABI** 决定。  
- 成员指针按下面 2.3 的 C 类型等价处理；`T&`、`T&&` 按 `T*` 处理。  
- 空类（若 C ABI 未规定）：size = align = 1。  
- `bool` 若 C ABI 有 `_Bool` 规则则跟随，否则 size = align = 1。  

POD 的 dsize、nvsize、nvalign 即其普通 size/align。

### 2.3 成员指针

#### 2.3.1 数据成员指针

- **表示**：`ptrdiff_t`，值为**该成员相对“基类型对象地址点”的字节偏移**。  
- **空指针**：偏移 `-1`。  
- 基类/派生类转换通过加减基类在派生类中的静态偏移完成；标准不允许成员指针在虚基关系上做这种转换，因此总是有静态偏移。  
- **已知缺陷**：某些转换可能得到 -1 的偏移，从而被误判为空指针。新平台建议采用不同表示（如偏移左移一位，用低位表示非空）。

#### 2.3.2 成员函数指针

- **通用布局**：等价于 `struct { fnptr_t ptr; ptrdiff_t adj; }`。  
- **非虚成员函数**：`ptr` 为函数指针，`adj` 为调用前需加到 `this` 的偏移。  
- **虚成员函数（标准表示）**：`ptr` 为 `1 + 虚表槽偏移` 的整型再转成函数指针形式，`adj` 仍为 this 调整。  
- **空指针**：`ptr` 为 null；32 位 ARM 表示中还需看 `adj` 的低位。  
- **32 位 ARM 等**：若函数指针低比特表示模式（如 Thumb），需使用** alternate representation**（`adj` 左移一位，低比特区分“函数指针”与“虚表偏移”）。  

调用时：先加 `adj` 得到调整后的 this，再根据 `ptr` 是函数地址还是虚表偏移，做直接调用或通过虚表调用。

### 2.4 非 POD 类类型布局（概要）

布局顺序大致为：

1. **初始化**：若为动态类，确定 primary base（若有），或在其偏移 0 放置虚表指针。  
2. **非虚部分**：按声明顺序分配 primary base（若未在步骤 1 处理）、其它非虚直接基类、非静态数据成员与未命名位域。空基类/空数据成员可尝试放在偏移 0 以节省空间（EBCO）。  
3. **虚基类**：按继承图顺序分配各虚基类（primary 与 indirect primary 已在前面处理过的不再重复分配）。  
4. **收尾**：处理 `[[no_unique_address]]` 等可能重叠成员，最后将 sizeof 向上取整到 align 的倍数。  

细节涉及：位域分配、类型冲突时偏移递增、dsize/nvsize/nvalign 的更新等，见规范正文。

### 2.5 虚表布局（Virtual Table Layout）

下面用“从零开始”的方式说明虚表是什么、为什么需要、以及 ABI 里具体长什么样，方便 C++ 小白理解。

#### 2.5.0 先弄懂：虚函数和虚表在解决什么问题

- **虚函数**：用 `virtual` 声明的成员函数。通过**基类指针或引用**调用时，运行时会根据**实际对象类型**决定调用哪一个实现（派生类可以重写/override）。
- **多态**：同一句“调用虚函数”的代码，在不同时刻可能调用不同函数，这就是**多态**。
- **编译器难题**：编译时往往只知道“指针是基类类型”，不知道它实际指向的是基类还是某个派生类，所以**无法在编译期固定要调用的函数地址**，必须在**运行时**根据对象查表再跳转。这张“表”就是**虚表（vtable）**。

可以这样类比：每个对象里藏着一张“菜单”（虚表），上面写着“第 1 个虚函数→实际函数 A，第 2 个虚函数→实际函数 B……”。调用虚函数时，编译器生成的是“按菜单第几项去调”的代码，而不是“直接调某个固定函数”的代码。

#### 2.5.1 对象里多了什么：虚表指针（vptr）

- 一旦类里有虚函数（或虚基类），编译器就会给**每个对象**在布局里加一个隐藏的指针成员，通常放在对象开头，叫**虚表指针（vptr）**。
- 这个指针指向一张**表**，表里存的是“这个类”的虚函数信息（以及后面要说的偏移、RTTI 等）。同一类型的所有对象共享**同一张表**，所以不会每个对象都复制一整张表，只是每人持有一个指针。
- **重要**：对象里存的不是“表的最开头”，而是表中某一个固定位置，这个位置叫**地址点（address point）**。表里有些内容在地址点“前面”（低地址），有些在“后面”（高地址），这样设计是为了让不同角色的虚表可以共用同一块内存的不同视图。

#### 2.5.2 虚表里到底有什么：按顺序看每一项

可以把虚表想象成一块连续的内存，里面按**固定顺序**放下面这些东西（有虚基类时前面还会多 vcall/vbase 相关项）。对象里的 vptr 指向的是下面说的“虚表地址点”位置。

1. **vcall offset**（有虚继承时才有，且在地址点之前）  
   - **用途**：当通过**虚基类**指针/引用调用虚函数、且该函数在更派生的类里被 override 时，需要把 `this` 从“虚基类子对象”调整到“真正定义该函数的派生类子对象”。vcall offset 就是“要加（或减）多少字节”的那个值。  
   - **小白理解**：虚基在内存里可能“飘”在对象中间，通过虚基调虚函数时，需要先根据这张表里存的偏移，把 `this` 指到正确的那一块再调。

2. **vbase offset**（有虚基类时才有，且在地址点之前）  
   - **用途**：从“当前子对象（即存着这份虚表指针的那块）”的地址，**加上**这个偏移，就能得到某个**虚基类子对象**的地址。每个虚基对应一个 vbase offset。  
   - **小白理解**：虚继承下，虚基只存在一份，可能不在对象开头。要访问虚基的成员或再通过虚基调虚函数，就要先算“虚基在哪”——vbase offset 就是用来算这个的。

3. **offset to top**  
   - **含义**：从“当前这个 vptr 在对象里的位置”到**整个完整对象（最派生对象）的起始地址**的字节差，类型是 `ptrdiff_t`（可正可负）。  
   - **用途**：例如 `dynamic_cast` 需要知道“完整对象从哪里开始”，再查 RTTI 判断类型。从任意基类子对象出发，用当前 vptr 对应的虚表里的 offset to top，就能反推完整对象首地址。  
   - **常见值**：对“完整对象自己的主虚表”，这个值一般是 0（vptr 就在对象顶）；对“嵌在派生类里的基类”的次级虚表，往往是负数（因为基类子对象在完整对象后面，即低地址方向）。

4. **typeinfo 指针**  
   - **用途**：指向该类型的 RTTI 信息（用于 `typeid`、`dynamic_cast`、异常匹配等）。  
   - 有虚函数的类（多态类型）这里是一个有效指针；非多态类可以是 0。

5. **虚表地址点**  
   - 也就是**对象里 vptr 真正指向的那个位置**。ABI 规定这里要满足指针的对齐要求。上面说的 offset to top、typeinfo 等都在地址点附近；再往后就是虚函数指针。

6. **虚函数指针**  
   - 按类里**虚函数的声明顺序**依次排列。每个槽位要么是“最终要调的那个函数”的地址，要么是一小段** thunk**（先调整 `this` 再跳转到真实函数）。  
   - **虚析构**占两个槽：一个用于“只析构不 delete”（complete object destructor），一个用于“析构完再 delete”（deleting destructor）。

小结：**vptr → 虚表地址点 → 前面是 offset/typeinfo，后面是一串虚函数指针**；有虚基时前面还会多 vcall/vbase 相关的槽。

#### 2.5.3 主虚表 vs 次级虚表；虚表组

- **主虚表**：最派生类“自己”用的那张表，也就是对象**顶部**那个 vptr 指向的表。里面包含该类的所有虚函数（以及前面说的 offset、typeinfo 等）。
- **次级虚表**：当这个类作为**基类**被继承时，派生类对象里会有“基类子对象”。每个需要 vptr 的基类子对象都有自己的 vptr，指向的是一张“基类在派生类中的表”，即 **Base-in-Derived** 的虚表，这类表叫**次级虚表**。  
  - 次级虚表的布局和“基类单独存在时的虚表”一致，这样通过基类指针调虚函数时，代码只要按固定偏移查表即可。但表里的**内容**会针对“被嵌在派生类里”做调整：例如 offset to top 指向的是完整派生对象顶，typeinfo 可以是派生类，被 override 的虚函数槽会填成派生类的实现（或 thunk）。
- **虚表组（vtable group）**：主虚表 + 所有次级虚表在内存里连在一起，形成一组。生成代码时，往往把整组放在一起，通过“主虚表地址 + 固定偏移”访问各次级虚表。

**为什么要分主/次级？**  
因为一个派生类对象里可能有多块“带 vptr 的”区域（最派生一块，每个带虚函数的基类子对象各一块）。每块在调用虚函数时，都要用**自己的**这张表，才能保证：  
- 用派生类指针调 → 用主虚表，看到派生类的 override；  
- 用某个基类指针调 → 用该基类对应的次级虚表，看到的也是正确的 override 和正确的 offset/typeinfo。

#### 2.5.4 虚表按“继承形状”分成的几类

ABI 按类的继承结构，把虚表分成几类，每类要放哪些内容、有没有次级表，规则略有不同：

- **Trivial（无虚）**：没有虚函数、也没有虚基类。没有虚表，对象里也没有 vptr。
- **Leaf（叶子类）**：自己有虚函数，但没有继承别人的虚函数，也没有虚基类。只有一张主虚表，里面是 offset to top、typeinfo、虚函数指针。
- **仅非虚基**：只继承非虚基类（可能有多个）。有主虚表 + 若干次级虚表（每个带虚表的非 primary 基类一张）。主虚表里先是一份“primary base 的虚表内容”（可能被派生类覆盖），再追加派生类自己的虚函数。
- **仅虚基**：只有虚基类（且没有“近乎空”的 primary）。主虚表里要有 vbase offset 等；每个虚基对应一张次级虚表，里面可能有 vcall offset。
- **复杂**：既有非虚基又有虚基，或存在“近乎空”的虚基等。规则是上面“仅非虚基”和“仅虚基”的组合，主表、次级表、vcall/vbase 都按组合规则来。

**Primary base**：若派生类的**第一个**非虚、且带虚表的基类满足“近乎空”（只有 vptr，没有其它数据），则它可以和派生类**共用同一块 vptr**，即共享主虚表的前半段，这样省一个指针。这时我们说它是派生类的 primary base。

#### 2.5.5 小结：谁有一份表、表里有什么

- 每个**有虚函数或虚基类的类**，都对应**一组**虚表（主表 + 若干次级表）。  
- 同一**最派生类型**的所有对象，其各个 vptr 指向的是**同一组**虚表（同一条主表、同一批次级表）。  
- 表里顺序大致是：可选的 vcall/vbase 偏移 → offset to top → typeinfo → **地址点** → 虚函数指针；虚析构占两个槽。  
- 通过 vptr 找到的是**地址点**，不是整张表的最开头，所以表里可以有“负偏移”和“正偏移”的项。

### 2.6 对象构造期间的虚表（VTT）

这一节解释：**在构造和析构过程中，为什么有时不能直接用“正常”的虚表，而要再用一套“构造虚表”和 VTT”**。目标是让 C++ 小白也能理解“构造顺序”和“虚基偏移”带来的问题。

#### 2.6.0 构造时对象是“半成品”，类型在变

- 构造顺序是：先基类子对象，再成员，最后派生类自己的 body。在**执行某个基类的构造函数**时，当前对象里只有：该基类及其更底层基类已经构造好的部分，派生类部分还没构造。  
- C++ 规定：在基类构造函数体内，`typeid`、`dynamic_cast` 和**虚函数调用**看到的应该是**当前正在构造的这个基类**，而不是最终的最派生类。这样才安全，也不会用到还没初始化的派生类成员。  
- 因此，在进入每个基类构造函数时，编译器会把**该基类子对象**里的 vptr 设成指向“代表当前基类”的虚表，这样虚调用和 RTTI 就符合语义。

**没有虚继承时**：每个基类子对象在派生类里的偏移是固定的，所以用“Base-in-Derived”的**正常**次级虚表即可。构造器里把 vptr 指到这张表，就能正确调当前基类的虚函数，且 offset to top、typeinfo 等也是对的。

#### 2.6.1 虚继承带来的问题：同一基类，两种布局

- **虚基类**在最终对象里只存在**一份**，位置由最派生类决定，可能不在“基类子对象”的头部，甚至不在该基类“单独存在时的布局”里。  
- 举例：类 `B` 虚继承 `A`。当 `B` **单独**作为完整对象时，`A` 在 `B` 里有一个固定偏移。但当 `B` 作为更派生类 `D` 的基类时，`A` 在 `D` 里的偏移可能和“`A` 在单独 `B` 里的偏移”**不一样**（因为 `D` 可能还有别的基类，虚基 `A` 被放在别处）。  
- 于是就有两种“同一类型”的布局：  
  - **B 作为完整对象**：虚表里记录的 vbase offset 是“从 B 对象顶到 A”的偏移。  
  - **B 作为 D 的基类（B-in-D）**：虚表里记录的应该是“从 D 对象顶到 A”的偏移（或从 B-in-D 的 vptr 到 A 的偏移），否则在 D 的构造过程中，通过 B 的 vptr 去访问虚基 A 就会找错位置。

因此，在**构造 D 的过程中**，当执行到 **B 的构造函数**时：  
- 对象实际上是“D 的半成品”，虚基 A 已经在 D 的某个位置；  
- 此时 B 子对象里的 vptr 若还指向“B 单独存在时的虚表”，那里的 vbase offset 是针对“B 完整对象”的，在 D 里就是错的。  
- 所以这段时间内，B 的 vptr 必须指向一张**专门为“B-in-D 在构造阶段”准备的虚表**，里面的 vbase offset（以及依赖它的 vcall 等）要按**完整对象 D** 的布局来填。这张表就是**构造虚表（construction vtable）**。

#### 2.6.2 构造虚表里装的是什么

- **构造虚表**在布局上和“正常”的 Base-in-Derived 虚表类似，但**内容**针对“正在被构造的完整对象”：  
  - 虚函数指针、offset to top、typeinfo 仍然表示**当前基类 B**（这样虚调用和 RTTI 看到的是 B）。  
  - **虚基偏移（vbase offset）** 以及依赖它们的 vcall 等，则按**完整对象 D** 的布局来填，这样在 B 的构造函数里通过虚基做虚调用或访问虚基成员时，能正确找到 D 里的那一份虚基。  
- 所以：构造虚表 = “类型信息还是基类的” + “布局信息（虚基位置等）是完整派生对象的”。

#### 2.6.3 VTT：构造时该用哪张表，由它说了算

- 最派生类 D 的构造函数负责按顺序调用各个基类构造函数。每个基类构造需要知道：“你现在是 D 的一部分，请把你的 vptr 设成我给你的这张表”。  
- **VTT（Virtual Table Table）** 就是 D 生成的一张**表，表里存的是虚表指针**。D 的构造函数会根据构造顺序，从 VTT 里取出对应的项，传给各个基类构造函数（或用来设置各子对象的 vptr）。  
- 具体传的是“VTT 里某一段”的起始地址（即 sub-VTT），基类构造函数再按约定从这段里取：  
  - 主 vptr 该指向哪（主虚表或某张构造虚表）；  
  - 若有多个 vptr（例如该基类内部还有带虚表的子对象），它们各自该指向 VTT 里的哪一项。  

这样，**“构造时用哪张虚表”完全由 VTT 决定**：有的阶段用构造虚表（有虚基的基类），有的阶段用正式的 D 的虚表组（例如 D 自己主体初始化时，或没有虚基的基类）。

#### 2.6.4 VTT 里大致有什么（不背也行，理解即可）

- **主虚表指针**：D 的完整对象最终使用的主虚表（或构造末尾要设成的表）。  
- **次级 VTT**：若某个基类自己也有虚基，它构造时需要自己的“子 VTT”，所以 VTT 里会递归包含“B-in-D 的 sub-VTT”。  
- **各基类子对象的虚表指针**：按继承图顺序，每个需要 vptr 的基类（或虚基）在构造阶段应使用的虚表地址——可能是构造虚表，也可能是 D 的正式虚表组中的某一张。  
- **虚基的 VTT**：虚基类可能也要跑构造函数，它们也需要自己的 sub-VTT，所以 VTT 里还会有一段“给各虚基构造用”的虚表指针。

析构是构造的逆过程，同样在析构基类子对象时，会有一段时间对象“退化成”基类，vptr 也要切回合适的表，因此也会用到 VTT 和构造虚表（或与构造对称的析构用表）。

#### 2.6.5 为什么构造虚基时不能乱用 vptr

- 在有多个虚基、且它们可能共享同一 vptr 位置时（例如“近乎空”的虚基和派生类共用一个 vptr），在构造的某一时刻，这个 vptr 可能指向的是**另一个**基类的虚表（例如正在构造 T，vptr 指向的是 T 的构造虚表），而不是当前要构造的虚基 U 的虚表。  
- 因此，在**构造虚基 U** 时，不能假设“对象顶上的 vptr 能用来找 U”；必须通过 VTT 或其它约定，单独为 U 子对象设置正确的 vptr，或单独计算 U 的位置。规范里特别提醒：完整对象构造器在构造虚基时，要小心不要误用可能指向别家虚表的 vptr 来定位当前虚基。

#### 2.6.6 小结（2.6）

- **构造/析构过程中**，基类子对象的 vptr 会指向“代表当前基类”的虚表，这样虚调用和 RTTI 符合标准语义。  
- **有虚继承时**，基类若单独存在，其虚表里的虚基偏移是“单独对象”的布局；但该基类嵌在派生类里时，虚基在完整对象中的位置可能不同，所以需要**构造虚表**，里面用完整对象的布局填 vbase offset 等。  
- **VTT** 是一张“虚表指针的数组/表”，由最派生类生成，在构造（和析构）时告诉各个基类构造函数“该把自己的 vptr 设成哪张表”。  
- 这样，C++ 小白只要记住：**构造时类型是“半成品”，要用专门为构造阶段准备的虚表（和 VTT），才能正确找到虚基和做虚调用**。

### 2.7 数组 new 的 Cookie

- `new T[n]` 在以下情况需要** cookie**（在数组元素前的一段头）：  
  - 元素类型 `T` 有非平凡析构函数，或  
  - 存在可匹配的、带 `size_t` 参数的类专属 `operator delete[]`。  
- Cookie 存元素个数；分配大小 = prefix_size + n * sizeof(T)，其中 prefix_size 考虑对齐与 cookie 大小。  
- `delete[]` 时通过 cookie 取回元素个数与分配起点，再调用析构并释放。

### 2.8 初始化守卫变量（Guard Variables）

- 函数内静态变量或具有 vague linkage 的静态数据成员（如类模板的静态成员）若需**动态初始化**，则有一个关联的 **guard 变量**（64 位）。  
- 首字节：0 表示未初始化，1 表示已初始化完成。  
- 用于保证只初始化一次，并与 3.3.3 的一次性构造 API（`__cxa_guard_acquire` / `__cxa_guard_release` / `__cxa_guard_abort`）配合。

### 2.9 运行时类型信息（RTTI）

- 用途：`dynamic_cast`、异常匹配、`typeid`。  
- **type_info**：标准接口（name、比较等）；实现中带 `__type_name` 等。  
- **类类型**：`__class_type_info`、单继承 `__si_class_type_info`、多继承/虚继承 `__vmi_class_type_info`，以及 `__base_class_type_info`（含 offset、虚/公开等标志）。  
- **指针类型**：`__pbase_type_info`（const/volatile/incomplete 等标志）、`__pointer_to_member_type_info`（带 context）。  
- **dynamic_cast**：通过 `abi::__dynamic_cast(sub, src_type, dst_type, src2dst_offset)` 实现，src2dst_offset 为可选的静态提示，用于加速常见 base-to-derived 转换。  
- 动态类的 type_info 与**主虚表**一起生成；基本类型的 type_info 由运行时库提供。

## 三、代码生成与 API（Code Emission and APIs）

### 3.1 函数

- 一般遵循**基础 C ABI** 的调用约定；C++ 特有部分：  
  - **this**：作为隐式第一个参数（非平凡返回值时可能在此之前还有返回槽）。  
  - **VTT 参数**：带虚基的基类子对象构造/析构函数有隐式 VTT 指针，紧跟在 this 之后。  
  - **非平凡参数**：按“调用者分配临时、传引用”方式；若析构非平凡，调用者在返回后负责析构。  
  - **非平凡返回值**：调用者传入隐式地址，被调方在该地址上构造返回值；若析构非平凡，由调用者负责析构。  
- 引用参数/返回值按“传指针”处理；空类型参数/返回值有特殊约定（如 Itanium 上 NaT 位处理）。

### 3.2 虚函数调用（Virtual Calls）

下面从“调用者”和“被调者”两边说明：**一次虚调用在机器层面是怎么完成的**，以及为什么需要 **thunk**（调整 this 再跳转）和 **vcall offset**。目标是让 C++ 小白也能看懂“虚表里存什么、调用时怎么做”。

#### 3.2.0 虚调用在做什么（高层流程）

当你写 `ptr->f()` 且 `f` 是虚函数时，编译器**不能**在编译期决定要调哪个函数（因为 `ptr` 可能指向基类或任意派生类）。所以生成的代码大致是：

1. 用 `ptr` 找到对象里的**虚表指针（vptr）**；
2. 通过 vptr 找到**虚表**，再按 `f` 在类里的**声明顺序**找到对应的**槽位**；
3. 槽位里存的是一个**函数地址**（或 thunk 的地址）；
4. 用当前 `ptr` 作为 `this`，**调用**那个地址。

也就是说：**谁声明了 f，就用谁那一块子对象里的 vptr 和虚表**；槽里指到谁，最终就调谁。下面分“调用者”和“被调者”说清楚细节。

#### 3.2.1 调用者（Caller）要做的四步

假设我们通过一个**静态类型为 B\*** 的指针调虚函数 `f`（B 可以是基类或最派生类）。ABI 规定调用者要做的事可以归纳为四步：

1. **选一个“声明了 f”的子对象 A**  
   在继承关系里，可能多个基类都声明了 `f`（例如 A 声明、B 也声明或继承）。调用者要选其中一个作为“用谁的虚表”。通常选**最终 overrider 所在的那个类**（即真正实现 `f` 的类 C）最省事，因为这样槽里往往直接就是 C::f，不用再经过 thunk 调整 this。但若手头已经有一个“声明 f 的基类 D”的指针，也可以选 D，用 D 的虚表调，这时槽里可能是一个“D-adjusting thunk”（见下）。

2. **把 B\* 转成指向该子对象的指针 `a`**  
   也就是算好“从当前 ptr 到子对象 A 的偏移”，得到 `a`（类型相当于 A*）。若选的就是 B 自己，则 `a == ptr`。

3. **用子对象 A 的 vptr 找到虚表，再按槽位取函数地址**  
   虚表里“第几个虚函数”是固定顺序（声明顺序），所以用 `a` 指向的对象里的 vptr 加上固定偏移，就得到“要调的那个函数”的地址。

4. **以 `a` 为 this，调用该函数**  
   即 `call 槽里存的地址`，第一个参数传 `a`（作为 this）。

**小白记忆**：先选“用哪块子对象的虚表”，把指针指到那块，查表取函数，再拿这块的指针当 this 去调。

#### 3.2.2 被调者（Callee）与“this 必须指到定义 f 的类”

- 真正实现虚函数 `f` 的往往是**派生类 C**（即 C::f 是 A::f 的 final overrider）。  
- C::f 的代码里会访问 C 的成员，所以 **C::f 期望的 this 必须指向“C 子对象”的起始地址**，而不是随便一个基类 A 的地址。  
- 因此：若我们通过 **A\*** 调 f（A 是基类，C 是派生），传进去的 this 是 A*，而 C::f 需要 C*。中间必须有人把 **A\* 调整成 C\***（加一个固定偏移），再跳转到 C::f。这段“调整 + 跳转”的小代码就是 **adjustor thunk**（或叫 A-adjusting entry point）。

**虚表里存什么**（对声明 f 的基类 A 而言）：

- 若 **A 就是定义 f 的类 C**（即 A = C）：槽里直接存 **C::f 的地址**（非调整入口），因为用 A* 调时，A* 就是 C*，不需要调整。
- 若 **A 是基类，C 是派生类**（A ≠ C）：槽里存的是 **“A-adjusting thunk”的地址**。这个 thunk 会：  
  - 把 this（当前是 A*）加上“A 子对象在 C 子对象中的偏移”，得到 C*；  
  - 再跳转到 C::f，并把 C* 作为 this 传进去。

这样，无论调用者选的是“最终 overrider 所在类”还是“某个声明 f 的基类”，只要按上面四步查表、传 this，最终都会调对 C::f，且 C::f 收到的 this 一定是 C*。

#### 3.2.3 虚继承时：vcall offset

- 当 **A 是虚基类**时，A 在对象里可能只存在一份，且位置由最派生类决定。通过 **A\*** 调 f 时，this 指向的是“虚基 A 子对象”；要调到 C::f，需要知道“从 A 子对象到 C 子对象”的偏移。  
- 这个偏移在**不同派生类**里可能不同（因为 A 在内存里的位置不同），所以不能写死在 thunk 里，而要**从虚表里读**。虚表里为每个这样的虚函数留了一个 **vcall offset** 槽位。  
- 因此，**虚基的“V-adjusting”入口**会：先用当前 this（V*）找到 V 的虚表，从表里取出 f 对应的 **vcall offset**，把 this 加上这个偏移得到 C*，再跳转到 C::f。  
- 若还有“道德上虚的”基类 M（M 是虚基 V 的基类，但不是虚继承），则 M-adjusting 会先做 M* → V*（固定偏移），再用 V 的虚表里的 vcall offset 做 V* → C*，再调 C::f。

**小白理解**：非虚继承时，基类在派生类里的偏移是编译期常数，thunk 里直接加常数即可；虚继承时，基类位置不固定，所以要从虚表里读“要加多少”，再调整 this。

#### 3.2.4 纯虚与删除的虚函数

- **纯虚函数**（未在派生类里 override 就被调用）：标准说是未定义行为。实现通常会在对应槽位里填 **`__cxa_pure_virtual`**（或它的包装），这样误调时会进到运行时的一个函数，一般会终止程序并报错。  
- **删除的虚函数**（`= delete`）被调用：同样未定义行为。槽位可填 **`__cxa_deleted_virtual`**，行为类似（终止程序等）。

这两个是 ABI 规定的**运行时占位 API**，编译器只需在生成虚表时把槽指过去即可。

#### 3.2.5 小结（3.2）

- **调用者**：选一个声明 f 的子对象 A → 把指针转成 A* → 用 A 的 vptr 查虚表取函数地址 → 以 A* 为 this 调用。  
- **被调者**：真正执行的是 C::f，它要求 this 是 C*。若通过基类 A* 调，则虚表槽里存的是 **A-adjusting thunk**：先做 this 从 A* 到 C* 的调整，再跳 C::f。  
- **虚继承**：基类位置不固定，要用虚表里的 **vcall offset** 做 this 调整。  
- 纯虚/删除的虚函数可指向 `__cxa_pure_virtual` / `__cxa_deleted_virtual`。

### 3.3 构造与析构 API（Construction and Destruction APIs）

这一节说明：**编译器在生成构造/析构、静态局部变量、数组 new/delete、全局对象析构时，会用到哪些运行时 API**。用“小白能懂”的方式讲清每个 API 在什么场景下被谁调用、做了什么。

#### 3.3.0 为什么需要这些 API

- 构造和析构的**顺序**、**只执行一次**、**异常时回滚**、**与动态库卸载顺序一致**等，单靠编译器生成“内联”代码很难统一处理。  
- 因此 ABI 规定一批**运行时库函数**（如 `__cxa_guard_acquire`、`__cxa_vec_new`、`__cxa_atexit` 等），编译器生成代码时**调用**这些函数，由运行时库保证语义正确。  
- 下面按“子对象构造与 VTT”“一次性构造（guard）”“数组构造/析构”“DSO 析构”几块分别说明。

#### 3.3.1 子对象构造与析构（和 VTT 怎么用）

- **完整对象**的构造函数（最派生类 D 的构造函数）负责：先按顺序调用各个**基类子对象**的构造函数，再初始化自己的成员，最后执行构造函数体。  
- 当某些基类**带有虚基类**时，基类构造期间需要用**构造虚表**（见 2.6）。因此 D 的构造函数必须能拿到 **VTT**，并把“属于每个基类的那一段”（sub-VTT）传给该基类的构造函数。  
- **VTT 怎么找到**：VTT 是编译器为类 D 生成的一块全局数据（数组），名字是 **mangled** 的（例如 `_ZTT1D` 这类）。D 的完整对象构造函数里，编译器会生成“取 VTT 符号地址”的代码，从而得到 VTT 的指针。  
- **怎么传**：调用基类 B 的构造函数时，若 B 需要构造虚表，则第二个参数（紧跟在 this 后面）传的是“VTT 中 B 对应的 sub-VTT 的起始地址”。基类 B 的构造函数收到这个指针后，按约定从里面取：  
  - 自己的主 vptr 该指向哪张表（第一项）；  
  - 若 B 内部还有带虚表的子对象，它们的 vptr 该指向 VTT 里的哪一项。  
  然后 B 的构造函数用这些值去**填写当前对象里各个 vptr**，这样在 B 的构造过程中，虚调用和虚基访问用的就是正确的构造虚表。  
- **不需要构造虚表的基类**（例如没有虚基类的基类）：可以不传第二个参数，基类构造函数用自己“完整对象”的虚表即可。  
- **析构**：是构造的逆过程；析构基类子对象时，同样会通过 VTT（或对称的析构用表）把 vptr 设回合适的表，保证析构过程中的虚调用和 RTTI 正确。

**小白记忆**：带虚基的继承下，最派生类通过“VTT”告诉每个基类构造函数“你该用哪张虚表”；基类构造函数用传进来的 sub-VTT 设置自己的 vptr（以及内部子对象的 vptr）。

#### 3.3.2 一次性构造（Guard）：静态局部与模板静态成员

- **场景**：函数内的 **static 局部变量**、或**类模板的 static 数据成员**，可能在不同翻译单元或多次调用中“逻辑上只应初始化一次”。标准要求：第一次执行到其声明时初始化，之后不再重复初始化。  
- **实现思路**：编译器为每个这样的变量关联一个 **guard 变量**（见 2.8），通常 64 位，其中**第一个字节**表示“是否已初始化”（0 = 未初始化，非 0 = 已初始化）。  
- **单线程简单做法**：进入作用域时，若 guard 首字节为 0，则执行初始化代码，然后把首字节设为非 0；否则跳过初始化。  
- **多线程且要保证只初始化一次**：需要用 ABI 规定的三个 API，在“做初始化”前后加锁/解锁：

  - **`__cxa_guard_acquire(&guard)`**  
    在真正做初始化**之前**调用。若返回 **1**，表示“当前线程抢到了初始化权”，接下来应执行初始化，然后必须调用 `__cxa_guard_release` 或（若初始化抛异常）`__cxa_guard_abort`。若返回 **0**，表示已有其他线程完成或正在做初始化，当前线程应跳过初始化、直接使用变量。  
  - **`__cxa_guard_release(&guard)`**  
    在初始化**成功完成**之后调用，把 guard 标成“已初始化”（通常设首字节为非 0），并释放 `acquire` 时拿的锁。  
  - **`__cxa_guard_abort(&guard)`**  
    当初始化**因异常失败**时调用，释放 `acquire` 时拿的锁，不把 guard 标成已初始化，这样下次进入还会再尝试。

**典型用法（伪代码）**：

```text
if (guard 首字节 == 0) {
  if (__cxa_guard_acquire(&guard) != 0) {
    try {
      执行初始化;  // 例如调用对象的构造函数
      用 __cxa_atexit 注册析构;
    } catch (...) {
      __cxa_guard_abort(&guard);
      throw;
    }
    __cxa_guard_release(&guard);
  }
}
// 此处可安全使用变量
```

**小白记忆**：静态局部/模板静态成员要“只初始化一次”，用 guard 变量 + 三个 API（acquire / release / abort）配合，既能单线程简单实现，也能做成多线程安全。

#### 3.3.3 数组的构造与析构（new[] / delete[]）

- **场景**：`new T[n]`、`delete[] p`。当 T 有非平凡构造/析构时，需要对**每个元素**分别调用构造函数或析构函数；若还有 cookie（见 2.7），还要在分配时预留 cookie、在释放时根据 cookie 取回元素个数和分配起点。  
- 编译器可以选择：要么自己内联展开循环调用每个元素的构造/析构，要么调用运行时提供的 **vector 构造/析构 API**。ABI 规定这些 API 的**接口**，实现由 libc++abi 等提供。

**分配 + 构造**（对应 `new T[n]` 的语义）：

- **`__cxa_vec_new`**：用默认的 `operator new[]` / `operator delete[]`，在分配的内存上预留 **padding_size**（放 cookie等），再对前 **element_count** 个元素依次调用 **constructor**。若某次构造抛异常，则对已构造的元素调用 **destructor**，再释放内存、重新抛出。  
- **`__cxa_vec_new2`** / **`__cxa_vec_new3`**：同上，但分配/释放函数由调用者传入（alloc/dealloc），便于与自定义 operator new[]/delete[] 配合。`_new3` 的 dealloc 带 `(void*, size_t)`，可传回大小。

**仅构造**（内存已有了，只对每个元素做构造）：

- **`__cxa_vec_ctor`**：给定数组首地址、元素个数、元素大小，对每个元素调用 constructor；异常时对已构造元素调用 destructor 再 rethrow。

**仅析构**（不释放内存）：

- **`__cxa_vec_dtor`**：对每个元素调用 destructor（逆序或顺序依实现）；若某个析构抛异常，会尽量把剩余元素析构完再 rethrow。  
- **`__cxa_vec_cleanup`**：同样对每个元素调用 destructor，但若析构抛异常则直接 `terminate()`，不继续析构其余元素（用于异常路径上的清理）。

**析构 + 释放**（对应 `delete[] p` 的语义）：

- **`__cxa_vec_delete`**：从 **array_address** 往前找到 cookie（按 padding_size），读出元素个数，对每个元素调用 destructor，再用 `operator delete[]` 释放整块内存。  
- **`__cxa_vec_delete2`** / **`__cxa_vec_delete3`**：同上，但用调用者提供的 dealloc。

**小白记忆**：`new[]`/`delete[]` 在涉及非平凡类型时，可以委托给 `__cxa_vec_*` 系列：分配+逐元素构造、逐元素析构、析构+释放；cookie 由 padding 和这些 API 共同约定。

#### 3.3.4 DSO 析构：全局/静态对象的析构顺序与 dlclose

- **场景**：全局对象、函数内 static 对象在程序退出时要**按构造的逆序**析构。而且，若程序用 `dlclose` 卸载某个动态库（DSO），该 DSO 里的全局对象也应在那时析构，而不是等到进程退出。  
- C 的 `atexit` 只能注册“进程退出时调用的函数”，且数量有限；也不能区分“是哪个 DSO 的析构”。所以 C++ ABI 规定一套 **atexit 的替代 API**，按“归属的 DSO”登记析构，并在程序退出或 `dlclose` 时按正确顺序执行。

**登记析构**：

- **`__cxa_atexit(f, p, d)`**  
  表示：当“与句柄 `d` 关联的 DSO 被卸载”时，在合适的时机调用 **f(p)**。  
  - `f`：析构时调用的函数（通常是析构函数的包装，会对 `p` 做一次析构）。  
  - `p`：要析构的对象的地址（或包装结构的地址）。  
  - `d`：**DSO 句柄**，一般用该编译单元链接器提供的 **`__dso_handle`**。  
  同一 DSO 内，后登记的会先执行（LIFO），这样就能实现“构造顺序的逆序”析构。

**触发析构**：

- **`__cxa_finalize(d)`**  
  - 若 `d != NULL`：执行所有“归属 DSO `d`”的、已通过 `__cxa_atexit` 登记的函数，并从未登记列表中移除（或标记为已执行）。  
  - 若 `d == NULL`：执行**所有**已登记的析构（程序退出时主程序会这样调一次）。  
  多次对同一 `d` 调用 `__cxa_finalize` 不会重复执行已执行过的登记项。

**链接器配合**：

- 每个会调用 `__cxa_atexit` 的编译单元（例如包含全局对象定义的 .o），链接器会提供一个符号 **`__dso_handle`**（指向该 DSO 内某段数据即可，只要每个 DSO 不同）。  
- 该 DSO 的 **FINI**（或 `.fini_array`）里应**首先**调用 **`__cxa_finalize(&__dso_handle)`**，这样在 DSO 被卸载（如 `dlclose`）时，会先执行本 DSO 已登记的所有析构，再执行其它 FINI。

**小白记忆**：全局/静态对象构造完后，用 `__cxa_atexit(析构包装, 对象地址, __dso_handle)` 登记；程序退出或 dlclose 时通过 `__cxa_finalize` 按顺序执行这些析构，保证顺序和 DSO 边界正确。

#### 3.3.5 其它（继承构造函数、构造顺序优先级）

- **继承构造函数**（`using Base::Base`）：形式上由基类构造函数初始化派生类，ABI 允许把这类逻辑抽成“继承构造函数”函数，参数与基类构造函数一致（虚基等有特殊规则），不要求实现必须这样做，但若实现抽成函数，需遵守相同语义。  
- **构造顺序优先级**：标准只规定同一翻译单元内全局对象的构造顺序；跨翻译单元的顺序未规定。部分实现支持 **`#pragma priority(n)`** 等扩展，用更小的 n 表示更早初始化；ABI 里用 `.priority_init` 段等描述，属可选实现细节。

#### 3.3.6 小结（3.3）

- **子对象构造/析构**：最派生类通过 **VTT** 的 mangled 名拿到 VTT，把各基类对应的 **sub-VTT** 传给基类构造函数；基类用其设置 vptr，从而在构造/析构阶段使用正确的虚表。  
- **一次性构造**：**guard 变量** + **`__cxa_guard_acquire`** / **`__cxa_guard_release`** / **`__cxa_guard_abort`**，用于静态局部、模板静态成员等的“只初始化一次”（可做成线程安全）。  
- **数组**：**`__cxa_vec_new`** / **`__cxa_vec_new2`** / **`__cxa_vec_new3`** 做分配+逐元素构造；**`__cxa_vec_ctor`** / **`__cxa_vec_dtor`** / **`__cxa_vec_cleanup`** 做仅构造/仅析构；**`__cxa_vec_delete`** 等做析构+释放；与 cookie 约定一致。  
- **DSO 析构**：**`__cxa_atexit(f, p, d)`** 登记“DSO d 卸载时调用 f(p)”；**`__cxa_finalize(d)`** 执行登记项；链接器提供 **`__dso_handle`**，FINI 中调用 **`__cxa_finalize(&__dso_handle)`**，保证全局对象析构顺序与 dlclose 行为正确。

### 3.4 Demangler API

- `abi::__cxa_demangle(mangled_name, buf, n, status)`：将 mangled 名解成可读字符串。  
- 可接受外部名（如带 `_Z` 前缀）或内部 NTBS 类型名。  
- 返回值为新分配或传入的 buf；status 表示成功(0)、分配失败(-1)、非法名(-2)、非法参数(-3) 等。

## 五、链接与对象文件（Linkage and Object Files）

### 5.1 外部名（Name Mangling）

- **目的**：在目标文件中唯一表示 C++ 实体（函数重载、命名空间、类、模板等），以便链接与符号解析。  
- **格式**：`_Z` + `<encoding>`；encoding 包含名字与（对函数）类型信息。  
- **名字**：全局/命名空间/类内名字、嵌套名、模板名、局部名（带函数 encoding + 可选的 discriminator）等。  
- **类型**：内置类型单字母码（如 `i`=int, `v`=void）、指针/引用、函数类型（F...E）、类/枚举名、数组、成员指针、模板参数等。  
- **表达式**：模板参数中的常量、`sizeof`/`alignof`、`decltype` 等会编码为表达式，使用前缀遍历的语法树编码。  
- **压缩**：通过 substitution（`S` + seq-id) 和常用缩写（如 `St`=std, `Ss`=std::string）缩短名字。  
- **特殊名**：虚表 `TV <type>`、VTT `TT <type>`、type_info `TI`/`TS`、guard 变量 `GV`、构造/析构（C1/C2/C3/D0/D1/D2 等）、thunk（T h/v offset + base encoding）等。  

Mangling 语法非常细，涵盖模板、lambda、闭包类型、ABI tag、异常规范等，详见规范第 5.1 节及 [mangling table](https://itanium-cxx-abi.github.io/cxx-abi/abi-mangling.html)。

### 5.2 Vague Linkage

- 对**可在多个翻译单元出现但 ODR 要求单一定义**的实体，规定**在何处生成**以及如何**去重**。  
- **Inline 函数**：每处引用都可生成一份，通过 COMDAT 等机制去重，以函数 mangled 名为标识。  
- **静态数据**：与 inline 或模板相关的静态数据/字符串用局部名 mangling，并在 COMDAT 中以该名标识；guard 变量若与数据同组可一起放入 COMDAT。  
- **虚表**：若有 **key function**（第一个非纯虚、非 inline 的定义），虚表在包含 key function 定义的单元中**唯一**生成；否则为 vague linkage，每处引用都可生成，COMDAT 去重。模板类虚表每实例化处都可能生成，COMDAT 去重。  
- **type_info**：与虚表同单元唯一生成，或按类型/模板规则 vague linkage；基本类型由运行时库提供。  
- **构造/析构**：与普通函数类似；若为 inline 或模板实例，按引用生成，COMDAT 去重。  
- **模板实例**：类模板实例化时，虚表、VTT、静态成员、虚/非虚成员函数等按规范在“实例化处”或“引用处”生成，并用 COMDAT 标识。

### 5.3 Unwind 表位置

- Itanium 实现将 unwind 表放在 `SHT_IA_64_UNWIND` 段，并与相关代码链接；链接器将 unwind 表、unwind 信息与代码放在同一 text 段，由 `PT_IA_64_UNWIND` 程序头标识。

## 六、学习要点小结

1. **数据布局**：POD 跟 C ABI；非 POD 先定 primary base 与虚指针，再按顺序排基类与成员，最后排虚基；空基优化、位域、尾部 padding 有细致规则。  
2. **成员指针**：数据成员指针为偏移（-1 表示空）；成员函数指针为 (ptr, adj)，虚函数时 ptr 可编码为虚表偏移，调用时需区分解法。  
3. **虚表与 VTT**：虚表包含 vcall/vbase 偏移、offset-to-top、typeinfo、虚函数指针；构造阶段用 VTT 指向的构造虚表以保证虚基和虚调用正确。  
4. **调用约定**：this、VTT、非平凡参数/返回值、引用、空类型均有约定；虚调用通过虚表 + 可能的 thunk 完成。  
5. **RTTI**：与虚表一起生成，支持 dynamic_cast、typeid、异常匹配；有单继承/多继承/虚继承及指针、成员指针的 type_info 层次。  
6. **Mangling**：`_Z` + 名字与类型编码，支持嵌套名、模板、表达式、lambda、压缩与缩写，保证 ODR 与链接一致。  
7. **Vague linkage**：inline、模板、虚表、type_info、构造/析构等“可多定义”实体，通过 COMDAT 与 key function 等规则确定唯一或一致定义。
